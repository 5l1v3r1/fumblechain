In this lesson we will talk about what's in a block, including the header and the list of transactions.

A blockchain is composed of blocks. Each blocks contains transactions. However, for the system to work, slightly more information than that needs to be stored in each block. We will now describe what additional information is stored inside of blocks and why that is the case.

Blocks usually contain a block number, sometimes called block height. This number is incremented by one for every new block added to the chain. This is useful because it directly tells us the block number. Assuming that we are working with the very latest block, its number tells us how many blocks are in the chain.

The hash of the previous block is used to link the block to the previous block. Of course, blocks also have a hash, so that the next block's previous hash can be set to this block's hash to enlarge the chain.

The timestamp at which the block has been created is also stored in a block. This is useful to store the date and time at which transactions have been performed. Especially since transactions do not have a timestamp, usually. Blocks have a timestamp. Every transaction in the same block is considered to have happened at the same time.

In the case of a Proof of Work blockchain, a nonce is added to the block structure. The nonce is a random value that miners successively change to make the block hash change so that it is less than the target hash required for the block to be successfully mined.

Finally, the transactions are stored in a block. They are, however, stored in a special structure and not in a simple list. This special structure used for storing transactions is usually a Merkle Tree. This structure has some properties that blockchains can benefit from to increase performance, such as:

  • It computes a hash of transaction hashes, grouped 2 by 2. That means that the root hash can be computed iteratively, as transactions are received by network nodes.
  • It allows for quickly proving the integrity of all the transactions in a block.
  • It allows for efficient comparison of the list of transactions in a block. Indeed, if the root hashes do not match, one can request the two sub hashes and compare those, and so on so we only have to transfer few hashes over the network and not the full transactions themselves to perform a comparison.

In summary, the block header is composed of:

  • block number
  • previous block hash
  • timestamp
  • nonce
  • merkle tree root hash
That concludes this lesson about block contents.
